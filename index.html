A Korean commercial bank's loan operations team processes incoming applications across mixed formats and channels. Each one must clear document intake → KYC/AML compliance → credit risk scoring → 결재 의견 (FSS-aligned approval recommendation). Below: four pain points specific to Korean financial services, and how this demo addresses each.

07 Guardrails — 4 Layers Wired 가드레일 4계층 — 실제 구현 방식

All four layers exist in the codebase, but not via a uniform decorator API. Three use manual hooks placed at the right Flow stage; one uses CrewAI's official Task(guardrail=...) parameter. Each layer emits a distinct OTel event family.

LAYER 1 · INPUT PII main.py:254 · guardrails/input_pii.py
입력 PII 스캔
Scans document_path text and masks regulated identifiers before any LLM ever sees them.
scan_and_mask_input(document_path)
manual hook in @listen input_pii_scan
OTel: guardrail.l1_input_pii.*
LAYER 2 · TOOL RBAC triage_crew.py:23-35 · guardrails/tool_rbac.py
도구 단위 RBAC
Wraps every tool's _run with a persona allow-list check — auditor blocked at every tool, loan_officer + compliance pass through. The shared-agent RBAC mechanism for this project.
wrap_tools_with_rbac([...])
tool wrap at crew construction
OTel: guardrail.l2_tool_rbac.*
LAYER 3 · HALLUCINATION main.py:304-306 · guardrails/hallucination.py
환각 검증
Scores each sub-agent output (intake / kyc / risk) for hallucination and gates the run. Aggregated post-kickoff.
score_hallucination() + hallucination_gate()
manual hook post-crew().kickoff()
OTel: guardrail.l3_hallucination.*
LAYER 4 · OUTPUT POLICY triage_crew.py:101 · 110 · 119 · 127 · guardrails/output_policy.py
출력 정책 (RRN · 전화 · 계좌 · 통화)
Regex checks for resident registration numbers, phone numbers, account numbers, and non-KRW currency in every task's final output.
Task(guardrail=policy_guardrail, ...)
canonical API CrewAI auto-retries on False
OTel: guardrail.l4_output_policy.*
Wiring note. CrewAI's canonical surface for guardrails is Task(guardrail=fn) (auto-retry on False), @before_tool_call / @after_tool_call, and @before_llm_call / @after_llm_call from crewai.hooks. Only L4 uses the canonical API today; L1–L3 use manual hooks placed at the right Flow stage. Migration to the canonical hooks is on the post-PoC roadmap.

08 Shared Agent · Tool-Layer RBAC 공통 Agent · 도구 단위 권한

The same agent definition is shared across operators. RBAC is enforced at the tool layer — wrap_tools_with_rbac() wraps every tool's _run with a persona allow-list check at crew construction time. One agent definition, per-call permission context.

MECHANISM · 메커니즘
One definition, persona-scoped tool whitelist
from loan_application_triage.guardrails import wrap_tools_with_rbac

# at crew construction (triage_crew.py:23-35)
(document_parse_mcp_tool, korean_doc_extractor,
 kyc_lookup_tool, sanctions_list_tool, credit_score_tool) = (
    wrap_tools_with_rbac([
        document_parse_mcp_tool, korean_doc_extractor,
        kyc_lookup_tool, sanctions_list_tool, credit_score_tool,
    ])
)

# wrap_tools_with_rbac patches each tool's _run
# to call rbac_check_tool_call(tool_name) — which
# reads get_active_persona() and returns a denial
# JSON to the LLM if persona.tools doesn't list it.
Active persona is read at call time via get_active_persona() (sources the persona from the OPERATOR_PERSONA env var, set by the JWT validation step in begin). The wrap happens once at module import; per-call enforcement is automatic for every _run invocation thereafter. Denied calls return a JSON error to the LLM rather than raising — preventing hallucination of fake results.
PROPERTIES · 속성
  • No per-user agent re-instantiation
    사용자마다 에이전트를 새로 만들지 않음 — 정의는 하나
  • Tool-call denial is RBAC's main enforcement point
    권한 분리의 핵심은 도구 호출 단계의 차단
  • Each denied call emits a distinct OTel event
    guardrail.l2_tool_rbac.deny
  • Personas defined in auth.py · tool wrap in guardrails/tool_rbac.py
    페르소나 정의 + 도구 화이트리스트가 한 파일에
Refactor candidate. Tool-wrapping monkey-patches _run, which is fragile against CrewAI internal changes. Canonical refactor: replace with @before_tool_call hook reading context.tool_name + active persona, returning False to block. Same OTel event semantics, stable across CrewAI versions.

09 Email-Approval Pause / Resume 이메일 승인 일시정지·재개

HITL is more than queueing — the Flow actually pauses mid-run, sends a tamper-proof email to the reviewer, and resumes when they click. Built into the route_for_human_review stage.

MECHANISM · 메커니즘
3 components, 1 flow stage
# Inside route_for_human_review (main.py:333-418)

# 1) Boot webhook (daemon thread, port 8765)
from .approval_webhook import start_in_background, wait_for_verdict
webhook_info = start_in_background(port=8765)

# 2) Send HMAC-signed email via macOS Mail.app
from .email_approval import send_approval_email
send_approval_email(applicant, kyc, risk, routing, audit_run_id)

# 3) Poll for verdict file (300s timeout)
verdict = wait_for_verdict(audit_run_id, timeout_seconds=300)
if verdict is None:
    state.hitl_routing["reviewer_verdict"] = "timeout"
else:
    state.final_decision = "approved" if verdict["verdict"] == "approve" else "rejected"
FastAPI webhook runs as a daemon thread in the same Python process. send_approval_email uses osascript to drive macOS Mail.app — no SMTP credentials, no API keys, just the Mail account the operator is already signed in to. wait_for_verdict polls output/verdict_<run_id>.json for up to 300 seconds.
TAMPER-PROOF MAGIC LINKS · 변조 방지 매직 링크
HMAC-SHA256 over (run_id, verdict)
# email_approval.py
def sign_token(audit_run_id, verdict):
    msg = f"{audit_run_id}|{verdict}".encode()
    return hmac.new(_secret(), msg, sha256).hexdigest()

approve_link = f"{base_url}/approve?run_id={rid}"
              + f"&token={sign_token(rid, 'approve')}"

# approval_webhook.py — verify on inbound click
@app.get("/approve")
def approve(run_id: str, token: str):
    if not verify_token(run_id, "approve", token):
        raise HTTPException(403, "invalid token")
    record_verdict(run_id, "approve")  # idempotent
Approve/Reject buttons in the email each carry a different HMAC signature. The webhook verifies before recording, so a captured link can't be tampered to flip the verdict. First click wins — subsequent clicks are idempotent (same payload returned).
REVIEWER ROUTING · 검토자 라우팅
6 routing rules, prioritized by failure mode
  • low_conf_identity → KYC 검증팀 시니어 (3h SLA)
    legal_name or entity_type confidence dominant
  • low_conf_credit → 신용분석팀 리드 (4h SLA)
    requested_amount_krw or declared_credit_grade dominant
  • low_conf_docs → 여신 서류심사역 (6h SLA)
    purpose or submitted_documents dominant
  • consumer_high_risk → 소매여신 위험관리역 (2h SLA)
    entity_type == "개인"
  • high_amount_corporate → 기업금융본부장 (8h SLA)
    requested_amount_krw ≥ 10억 KRW
  • default → 여신심사부 시니어 심사역 (4h SLA)
    otherwise (mid-amount corporate/SME)
PAUSE/RESUME SHAPE · 일시정지·재개 형태
In-process today, persisted in production
# Today (demo)
Flow process A:
  ├─ route_for_human_review (blocking)
  │  ├─ webhook thread (same process)
  │  └─ wait_for_verdict (sleep poll)
  └─ verdict file written → resume

# Production refactor
Flow process A:                  Flow process B (resumes):
  ├─ route_for_human_review        ├─ load TriageState from store
  ├─ persist state to store        ├─ read verdict file
  └─ exit                          └─ continue Flow

@persist(store="redis://...")
class LoanApplicationTriageFlow(Flow):
    ...
Same Flow shape; only the persistence backend changes. In-process is the right shape for a 90-second demo (Sample 2 / 3 paths); production swaps to @persist + a state store so reviewers can take days, not seconds, without holding a Python process open.
Configuration. Webhook port: HITL_WEBHOOK_PORT (default 8765). Reviewer inbox: HITL_REVIEWER_EMAIL. Base URL embedded in links: HITL_APPROVAL_BASE_URL (default http://localhost:8765). Verdict file location: output/verdict_<run_id>.json. Timeout: HITL_WAIT_TIMEOUT_S (default 300s).

10 Why a Flow (and not just a Crew) 왜 Flow가 필요한가

Five reasons the Flow wrapper is justified — and what changed after the email-approval wiring went in.

REASON 01
조건 분기 / Conditional branching
UNCHANGED @router hitl_gate remains the only branch point — HITL vs auto.
REASON 02
크루 외부 단계 / Pre/post-crew steps
UNCHANGED SSO/JWT, OTel, LLM routing eval, PII scan, hallucination gating, serialization all belong outside the Crew.
REASON 03
단계별 타입 상태 / Typed state across stages
STRENGTHENED TriageState now carries reviewer_verdict + decided_at between email send and emit_final.
REASON 04
사람 결재 일시정지·재개 / Pause-and-resume for human review
UPGRADED route_for_human_review now blocks on wait_for_verdict() for 300s; webhook + HMAC token + verdict-file injection all in one Flow stage.
REASON 05
단계별 옵저버빌리티 / Observability per stage
UNCHANGED Each @listen gets its own span/trace event; email/webhook activity logged under route_for_human_review's span.

Nuance on stage 4: Today's setup is in-process pause/resume — the Flow blocks; the webhook runs in a daemon thread of the same Python process. True cross-process pause/resume would need Flow persistence (@persist or flow.kickoff_async() with a state store). For a 90-second demo, in-process is exactly right; for production, swap to persisted-state resume but keep the same Flow shape.
